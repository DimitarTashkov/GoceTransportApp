# План за Миграция и Оптимизация на ASP.NET Core Система (Google Cloud Run)

## 1. Архитектура и Docker Миграция (GCP Cloud Run)

Тъй като системата се състои от два отделни компонента (Web Сайт и Web API) и съществуваща външна база данни, архитектурата ще бъде разделена на две независими Cloud Run услуги.

### 1.1. Dockerization (Контейнеризация)
* **Задача:** Да се създадат два отделни `Dockerfile` файла – един за Web проекта и един за API проекта.
* **Изисквания:**
    * Използване на multi-stage build (напр. `mcr.microsoft.com/dotnet/sdk:8.0` за билдване и `mcr.microsoft.com/dotnet/aspnet:8.0` за рънтайм) за оптимизация на размера.
    * **Много важно за Cloud Run:** Приложенията трябва да слушат на порта, подаден от системната променлива `$PORT` (Cloud Run по подразбиране очаква порт `8080`).

### 1.2. Деплоймънт и Инфраструктура
* **Задача:** Подготовка за качване на имиджите в Google Artifact Registry и деплоймънт в Cloud Run.
* **Изисквания:**
    * Създаване на два сървиса в Cloud Run: `web-frontend` и `web-api`.
    * Настройка на мащабирането (Scaling): Минимален брой инстанции = 0 (за да е безплатно при липса на трафик).

## 2. Управление на Базата Данни и Свързаност (Критично)

Текущата база данни **не се мигрира** и остава на сегашния си хостинг/сървър. 

### 2.1. Конфигуриране на Environment Variables
* В настройките на **API проекта** трябва да се предвиди използването на Environment Variable за Connection String към външната база данни.
* В настройките на **Web проекта** трябва да се предвиди променлива (напр. `ApiSettings__BaseUrl`), която да сочи към URL адреса на API-то.

## 3. План за Функционални Подобрения (Backend & Architecture)

### 3.1. Устойчивост на мрежови грешки (Resilience)
* **Внедряване на Polly:** В Web проекта HTTP клиентите, които викат API-то, трябва да бъдат конфигурирани с `Polly`. Да се добавят политики за *Retry* (повторен опит при временен срив на API-то). Това е задължително, за да се предотврати краш на сайта при евентуален "Cold Start" (събуждане) на API контейнера в Cloud Run.

### 3.2. Оптимизация и Сигурност
* **Кеширане:** Внедряване на `IMemoryCache` в Web проекта за често достъпвани данни от API-то, за да се намалят излишните заявки.
* **CORS Policies:** API-то трябва да бъде конфигурирано да приема заявки от специфичния URL на Web проекта.
* **Rate Limiting:** Добавяне на ограничение на заявките в API-то за предотвратяване на злоупотреби.

## 4. План за UI/UX и Дизайн Подобрения (Frontend)

### 4.1. Модернизация на интерфейса (Bootstrap)
* **CSS Framework:** Да се запази и надгради **Bootstrap**, тъй като текущият проект вече е изграден с него. Да се обнови до последната стабилна версия (напр. Bootstrap 5.3), за да се ползват по-модерни компоненти.
* **Responsive Design:** Гарантиране, че Bootstrap класовете (grid системата) работят безупречно на мобилни устройства.
* **Dark Mode (Тъмна тема):** Внедряване на опция за превключване между светла и тъмна тема (Bootstrap 5.3 поддържа това нативно).

### 4.2. Подобряване на потребителското преживяване (UX)
* **Loading States (Състояния на зареждане):** Всяко действие, изискващо връзка с API-то, трябва да има визуален индикатор (Bootstrap Spinners). Това маскира забавянето при евентуален Cold Start на Cloud Run.
* **Error Handling (Обработка на грешки):** При грешка от API-то, потребителят трябва да вижда приятелски съобщения (Bootstrap Toasts или Alerts), а не системни грешки.
